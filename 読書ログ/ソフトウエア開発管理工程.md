1. チェックリスト
    - マトリクスチェックリスト
    
    |項目|内容|
    |:--|:--|
    |入力|チェックする処理の入力値を記述する|
    |出力|結果として出力するデータを記述する|
    |確認|結果の合格・不合格を記述する|
    |重要度|重要度をＡ／Ｂ／Ｃなどにレベル分けする|
    |正常／異常／境界|該当するテストがどのケースに該当するかを宣言する|
    |チェックNo|チェックする処理の入力値を記述する|
    
    - プログラムテスト仕様書

    |項目|内容|
    |:--|:--|
    |プログラム名|プログラム名称を記述する|
    |シート番号|連番を振る|
    |No|テスト項目の通し番号を振る|
    |テスト項目|テストする機能の概略を記述する|
    |入力データ|入力するデータを記述する|
    |出力データ|期待する実行結果のデータを記述する|

    |判定|合否判定を記述する|
    |確認日|テスト結果を確認した日付を記述する|

1. ソフトウエア開発の目標値
    - プログラミング
        - 新規  
        St = S3  
        S4 = S3 = S0  
        - 修正  
        St = S1 * n1 + S2 * n2 + S3  
        S4 = S0 + S2 + S3  
        
        |変数|名称|内容|
        |:--|:--|:--|
        |S0|開発ベースプログラムステップ|開発時にベースとなるプログラム全体のステップ数|
        |S1|修正ベースプログラムステップ|修正を行う場合、対象のプログラムの母体ステップ数合計|
        |S2|修正ステップ数|プログラムの修正のステップ数|
        |S3|新規ステップ数|新規作成のプログラムのステップ数|
        |S4|製品ステップする|開発が完了したときのステップ数<br>次回の開発のベースとなるステップ数|
        |St|作業ステップ数|新規・修正で発生する作業ステップ<br>実際に開発するステップ数ではないことに注意|
        |n1|修正に伴い修正ベースプログラムを見直すパーセンテージ|単純な修正ステップ数に、範囲や難易度を盛り込む値で、20～30％くらいが妥当|
        |n2|修正量に対する重み係数|修正するプログラムの潜在的な仕様や難易度の負荷を盛り込む重み係数|

    - テスト項目  
    Tt = St * Pk  
    
    |変数|名称|内容|
    |:--|:--|:--|
    |Tr|目標テスト項目件数|テストする件数の目標値|
    |St|作業ステップ数|前述|
    |Pk|プログラムの種類|プログラムの種類による重み係数|  
    
        - Pkの例
            - システム制御系　Pk = 4～10
            - デジタル信号関係のミドルウェア　Pk = 10～30
            - アプリケーション　Pk = 20～40

        - 目標テスト項目件数の正常・異常・境界の配分率  
        正常／異常／境界の割合で全体で100％になる
        - 目標テスト項目件数のテスト工程の配分率
            - 単体テスト　80％
            - 結合テスト　15％
            - システムテスト   5％
    
    - 不良摘出件数  
    Bt = St * Nb
    
    |変数|名称|内容|
    |:--|:--|:--|
    |Bt|不良不良目的値|不良を摘出する件数の目標値|
    |St|作業ステップ数|前述|
    |Nb|何件の不良摘出を行うかの件数|Nb = 30～40が妥当|

1. 品質管理
    - 品質管理の留意点
        - 製品の品質を確保する
        - 納期通りに納品する
        - 品質が悪いからといって担当者を責めてはいけない  
    - 品質の確保＝要件どおりにシステムが動作すること  
    そのために、単体テスト・結合テスト・システムテストを実施する
    - テスト進捗管理
    
    - テストにおける管理者の役割
        - 目標値を設定し、テスト消化・バグ発生・テストのチェックリストの件数を定量的に反映する
        - テスト工程で、テスト消化・バグ発生・バグ修正の件数を定量的に反映する
        - プログラムの修正の履歴管理をする
        - バグとテスト項目の消化を実績をフォロー
            - 遅れている要因を分析し、対策をする
            - 不良内容を分析し、早く品質の作りこみを行う
                - 不良の内容分析項目
                    - プログラム変更管理表から分析
                    
                    |項目|概略|
                    |:--|:--|
                    |モジュール|モジュール単位で不良件数が分かり、全体でどの担当者に不良が集中しているか分かる|
                    |不良形態|修正した不良を外部から見て、不良のレベルを分析する<br>不良の内容から、どういう傾向の不良があるかを分析する|
                    |不良要員|不良を作りこんだ要因を分析し、開発担当者の癖が分かり、指導できる|
                    |不良工程|どこの開発工程で作りこまれた不良かが分かる|
                
                    - 不良形態
                    
                    |不良形態|内容|
                    |:--|:--|
                    |デグレード|修正で、今まで動いていたプログラムが動かなくなった|
                    |修正不十分|修正で、今まで動いていたプログラムの一部が動かなくなった|
                    |新規不良|新規開発で作りこんだ不具合|
                    |潜在不良|変更に関係なく、以前から存在していた不具合|
                    |その他|上記のどれにも該当しない不具合|
                    
                    - 不良要因

                    |不良要因|内容|
                    |:--|:--|
                    |ケアレスミス|不注意で発生した|
                    |仕様書の不備|仕様書に不備・あいまいな表現があった|
                    |検討不足|方式を考えるときに考慮不足があった(例外処理など)|
                    |理解不足|内容がよく分からないままプログラミングしてしまった|
                    |ハードウエア不良対策|ハードウエアの不良に対し、ソフトウエアで対応する|
                    |その他|上記のどれにも該当しない|
